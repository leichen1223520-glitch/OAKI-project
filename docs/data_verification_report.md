### 2.3 数据预处理管道

**运行脚本**：`notebooks/run_fpcm_shenzhen.py`  
**调用方式**：
```bash
cd /workspace/OAKI-project
PYTHONPATH=. python3 notebooks/run_fpcm_shenzhen.py
```

**流程说明**：

```
原始数据 (DOCX Table 1)
        ↓ [手工读入 MONTHLY_DATA 列表]
月度原始指标
        ↓ [build_monthly_input()]
ModelInput 对象（FPCM模型输入）
  - Q_in = 处理水量(万m³) × 10000 / 当月天数  [m³/d]
  - E_total_monthly = 月用电量 kWh
  - 水质参数：深圳典型值代入（Level 2 模式）
    - COD_in=290.0 mg/L, COD_out=30.0, TN_in=38.0, TN_out=10.0
    - NH3N_in=30.0, NH3N_out=1.5, TP_in=4.5
  - T_water：各月典型温度（10月26°C → 2月14°C → 3月18°C）
  - 药剂投加量：= 单耗(kg/万m³) × 处理水量(万m³)
        ↓ [FPCM.run()]
ModelOutput 对象（年度碳排放，单位：kg CO₂eq/年）
  - E_Scope1_CO2eq: CH₄+N₂O 直接排放 [kg/年]
  - E_Scope2_CO2eq: 电力间接排放 [kg/年]
  - E_Scope3_CO2eq: 药剂+污泥 其他间接 [kg/年]
        ↓ [÷12 ÷1000 换算]
月度碳排放（tCO₂eq/月）
  - Scope1_tCO2 = E_Scope1_CO2eq / 12 / 1000
  - Scope2_tCO2 = E_Scope2_CO2eq / 12 / 1000
  - Scope3_tCO2 = E_Scope3_CO2eq / 12 / 1000
        ↓ [保存]
data/processed/fpcm_monthly_46plants.csv (JSON)
```

**⚠️ 方法论注意**：FPCM模型以"年度"为计算单位（内部将月电耗×12换算为年电耗，CH₄/N₂O计算基于 Q_in×365 的年度处理量）。输出的 `E_unit_kgCO2_m3` 定义为：

```
E_unit = E_total_CO2eq(年) / (Q_in × 365)  [kg CO₂eq/m³]
       = E_total_CO2eq(年) / (处理水量(万m³) × 10000 / 当月天数 × 365)
```

这与"月度CO₂总量 ÷ 月度处理水量"的简单计算在短月（2月）有约 **7.9% 的偏差**。两种定义对比：

| 月份 | 天数 | 模型E_unit（kgCO₂/m³）| 简单月度单位 | 差异 |
|------|------|---------------------|------------|------|
| 2025.10 | 31 | 0.4895 | 0.4803 | +1.9% |
| 2025.11 | 30 | 0.5132 | 0.5203 | -1.4% |
| 2025.12 | 31 | 0.5156 | 0.5059 | +1.9% |
| 2026.01 | 31 | 0.5322 | 0.5222 | +1.9% |
| 2026.02 | **28** | 0.5207 | **0.5656** | **-7.9%** |
| 2026.03 | 31 | 0.5049 | 0.4954 | +1.9% |
| **6月加权均值** | — | **0.5114** | **0.5114** | **0.0%** |

**结论**：6月加权平均值（0.5114）使用哪种方法均一致，无误差。单月值的差异是由于模型使用年化日均流量所致，在±22%的Level 2不确定性范围内可忽略。

### 2.4 FPCM 碳排放计算核心公式

**Scope 1（直接排放）**：

```
E_CH4 = TOW × MCF × B0 × GWP_CH4    [kgCH₄等效CO₂/年]
  TOW  = Q_in × 365 × BOD_in × f_boc / 1000  [kgBOD/年]
  MCF  = 0.028（先验均值，深圳适用）
  B0   = 0.21 kgCH₄/kgBOD
  GWP_CH4 = 28（AR5）

E_N2O_nit = TN_removed × EF_nit × 44/28 × GWP_N2O
E_N2O_denit = TN_removed × EF_denit_ref × g(COD/TN) × 44/28 × GWP_N2O
  GWP_N2O = 265（AR5）

E_Scope1 = E_CH4 + E_N2O_nit + E_N2O_denit  [kgCO₂eq/年]
```

**Scope 2（电力）**：

```
E_Scope2 = E_total_monthly × 12 × EF_grid   [kgCO₂/年]
  EF_grid = 0.5271 kgCO₂/kWh（广东省，2023年）
  vs 全国均值 0.5839（差异 9.7%，Scope 2 若误用将高估约 5,068 tCO₂eq/月）
```

**Scope 3（其他间接）**：

```
E_chem = 碳源月用量×12×EF_carbon + PAC月用量×12×EF_PAC + ...
  EF_carbon_source = 1.4 kgCO₂eq/kg（乙酸钠）
  EF_PAC = 0.9 kgCO₂eq/kg

E_sludge = W_DS × EF_disposal（好氧堆肥封闭：360 kgCO₂eq/tDS）

E_Scope3 = E_chem + E_sludge
```

## 第三部分：数据问题清单

### 问题 A（严重）：缺少 2026.04–2026.06 三个月数据

**描述**：用户提及 `data/raw/` 中有"2026年数据汇总（污水厂）(更新到6月）"文件，但仓库中不存在。  
**可能原因**：该文件为 xlsx/csv 格式，被 `.gitignore` 阻止纳入版本控制。  
**影响**：
- 当前分析覆盖 6 个月（秋冬季），无春夏季数据
- 论文中"深圳只有6个月数据"的限制性结论尚未解除
- 季节性规律分析仅基于 10月–3月，缺少 4–6月升温期数据

**解决方案**：
1. 将含 4–6 月数据的文件以 `.docx` 格式（如现有文件命名风格）放入 `data/raw/`
2. 或修改 `.gitignore` 去掉 `data/raw/*.xlsx` 条目，commit xlsx 文件
3. 然后运行 `notebooks/extend_to_june_2026.py` 完成扩展分析

### 问题 B（轻微）：月度单位碳排放的计算口径不一致

**描述**：模型内部使用 `E_unit = E_total(年) / (Q_in × 365)`，对短月（2月28天）计算所得单位排放（0.5207 kgCO₂/m³）比直接用"月度CO₂/月度水量"（0.5656）低约 7.9%。

**影响**：个别月份单位排放强度在论文中略有低估（仅限于2月）。6月加权均值不受影响。  
**建议**：在论文脚注中说明此计算口径差异，或在代码中改用月度计算：
```python
# 修复方案：月度单位碳排放的正确计算
unit_monthly = (out.E_total_CO2eq / 12 / 1000) / (vol_wan_m3 * 10000) * 1000  # kg/m³
```

### 问题 C（轻微）：平均单位能耗口径差异

**描述**：原始报告（docx）称"综合吨水电耗 0.401 kWh/m³"（加权均值），而论文 chapter_03b 表3B-3 写的是"6月均值 0.405 kWh/m³"（算术均值）。  
**验证**：
- 加权均值：41,374.53 万kWh / 103,300.23 万m³ = **0.4006 kWh/m³**  
- 算术均值：(0.352+0.396+0.409+0.419+0.450+0.399)/6 = **0.4042 kWh/m³**

**建议**：在论文表3B-3脚注中说明采用算术均值，并与原报告的加权均值对比。差异约0.4%，在±22%不确定性范围内可接受。
